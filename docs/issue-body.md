# Bug Report: Tycho SBOM plugin does not resolve embedded Bundle-ClassPath JARs whose pom.xml inherits groupId/version from parent

> **Paste this text into a new Eclipse Tycho GitHub issue.**  
> Repository: https://github.com/eclipse-tycho/tycho

---

## Environment

| Item | Value |
|---|---|
| Tycho version | 5.0.3 |
| `tycho-sbom-plugin` version | 5.0.3 |
| Maven version | 3.9.x |
| Java version | 21 |
| OS | Linux x86_64 (also reproduced on Windows) |

---

## Summary

The Tycho SBOM plugin (`tycho-sbom-plugin`, goal `generator`, parameter
`process-bundle-classpath=true`) does **not** emit a component entry for an
embedded OSGi `Bundle-ClassPath` JAR when that JAR's internal
`META-INF/maven/.../pom.xml` inherits `groupId` and/or `version` from a Maven
`<parent>` element instead of declaring them explicitly.

A JAR with an identical directory structure but an explicit `groupId`/`version`
in `pom.xml` **is** resolved correctly, demonstrating the resolver parses
`pom.xml` but cannot (or does not) fall back to `pom.properties`.

---

## Minimal reproducible example

Clone and build:

```bash
git clone https://github.com/janosch24/tycho-sbom-demo.git
cd tycho-sbom-demo
mvn -V -e -DskipTests clean verify
```

The `package` phase builds the p2 update-site **and** materializes an Eclipse
product installation (required by the generator – see *Steps to reproduce*).
The `verify` phase runs the SBOM generator against the installed product.

Then verify the SBOM gap:

```bash
grep -c "jzy3d-jdt-core" com.example.sbom.repository/target/sbom/bom.json \
  && echo "FOUND" || echo "MISSING — bug confirmed"
```

## Steps to reproduce

> **Important:** `tycho-sbom-plugin:generator` requires an *installed* Eclipse
> product layout (with `configuration/config.ini`), not a plain p2 update-site.
> A plain `eclipse-repository` update-site (`target/repository/`) does **not**
> contain `configuration/`, so the generator throws `NoSuchFileException`.
> The steps below use a `.product` file to trigger product materialization via
> `tycho-p2-director-plugin:materialize-products`.

1. Create an OSGi `eclipse-plugin` bundle.
2. Embed two JARs under `lib/jzy3d/` and list them on `Bundle-ClassPath` in
   `MANIFEST.MF`:
   - `com.diogonunes:JColor:5.2.0`
   - `org.jzy3d:jzy3d-jdt-core:2.2.0`
3. Build the bundle and include it in an `eclipse-repository` p2 update-site.
4. Add a minimal `.product` file to the `eclipse-repository` module referencing
   the bundle and `org.eclipse.osgi`. Tycho will automatically run
   `materialize-products` during the `package` phase, creating a real Eclipse
   installation directory under `target/products/<uid>/<os>/<ws>/<arch>/`.
   That directory contains `configuration/config.ini`.
5. Configure `tycho-sbom-plugin:generator` on the repository module, pointing
   `<installation>` at the materialized product (not the update-site):
   ```xml
   <configuration>
     <installation>${project.build.directory}/products/com.example.sbom.demo.product/linux/gtk/x86_64</installation>
     <process-bundle-classpath>true</process-bundle-classpath>
   </configuration>
   ```
6. Run `mvn verify`.
7. Confirm the installation was created:
   ```bash
   ls com.example.sbom.repository/target/products/com.example.sbom.demo.product/linux/gtk/x86_64/configuration/config.ini
   ```
8. Inspect the generated SBOM at `com.example.sbom.repository/target/sbom/bom.json`.

---

## Expected behaviour

Both embedded dependencies appear as components in the generated CycloneDX SBOM:

```json
{ "type": "library", "group": "com.diogonunes",  "name": "JColor",         "version": "5.2.0" }
{ "type": "library", "group": "org.jzy3d",        "name": "jzy3d-jdt-core", "version": "2.2.0" }
```

---

## Actual behaviour

Only `JColor` appears. `jzy3d-jdt-core` is **absent** from the SBOM.

---

## Difference between the two JARs

Both JARs contain `META-INF/maven/<groupId>/<artifactId>/pom.properties` and
`pom.xml`. The critical difference:

### JColor 5.2.0 – resolved ✅

`META-INF/maven/com.diogonunes/JColor/pom.properties`:
```properties
#Generated by Maven
groupId=com.diogonunes
artifactId=JColor
version=5.2.0
```

`META-INF/maven/com.diogonunes/JColor/pom.xml` (abbreviated):
```xml
<project>
  <groupId>com.diogonunes</groupId>   <!-- ← explicit -->
  <artifactId>JColor</artifactId>
  <version>5.2.0</version>            <!-- ← explicit -->
  ...
</project>
```

### jzy3d-jdt-core 2.2.0 – NOT resolved ❌

`META-INF/maven/org.jzy3d/jzy3d-jdt-core/pom.properties`:
```properties
#Generated by Maven
artifactId=jzy3d-jdt-core
groupId=org.jzy3d
version=2.2.0
```

`META-INF/maven/org.jzy3d/jzy3d-jdt-core/pom.xml` (abbreviated):
```xml
<project>
  <parent>
    <groupId>org.jzy3d</groupId>
    <artifactId>jzy3d-all</artifactId>
    <version>2.2.0</version>         <!-- groupId + version inherited from here -->
  </parent>
  <artifactId>jzy3d-jdt-core</artifactId>
  <!-- NOTE: no <groupId> and no <version> in this POM -->
  ...
</project>
```

> **Key observation:** `pom.properties` contains the full GAV (`groupId`,
> `artifactId`, `version`) in both cases. Only `pom.xml` is missing the
> direct declarations for `jzy3d-jdt-core`. The SBOM resolver appears to
> parse `pom.xml` but does not use `pom.properties` as a fallback when
> `groupId`/`version` are absent from the POM itself.
>
> This pattern (inheriting `groupId`/`version` from a Maven `<parent>`) is
> very common in the open-source ecosystem. Any published JAR built this way
> would be silently omitted from the SBOM.

---

## Hypothesis / Root cause

When the SBOM plugin resolves embedded JARs it reads `pom.xml` from the JAR's
`META-INF/maven/.../` directory. If `groupId` or `version` are not present as
direct child elements of `<project>` (but only in a `<parent>` block), the
resolver cannot complete the GAV because the parent POM is not available at
resolution time. The `pom.properties` file, which always contains the full GAV
as generated by Maven during the original build, is apparently not consulted as
a fallback.

**Suggested fix:** When `pom.xml` is missing `groupId` or `version`, the resolver
should fall back to reading them from the co-located `pom.properties`.

---

## Additional context

- The original library (`jzy3d-jdt-core`) is an unmodified artifact from Maven
  Central; the incomplete `pom.xml` is the artifact as published.
- The issue is not specific to this library; any embedded JAR whose build uses
  Maven parent-POM inheritance (a very common pattern) is affected.
- Inspect the JARs yourself with:

  ```bash
  # After running mvn verify once (so lib/ is populated):
  unzip -p com.example.sbom.demo/lib/jzy3d/jzy3d-jdt-core-2.2.0.jar \
    META-INF/maven/org.jzy3d/jzy3d-jdt-core/pom.xml
  unzip -p com.example.sbom.demo/lib/jzy3d/jzy3d-jdt-core-2.2.0.jar \
    META-INF/maven/org.jzy3d/jzy3d-jdt-core/pom.properties
  ```
